# 创建工单

引导用户创建一个结构良好的 Linear 工单，包含验收标准和实施上下文，可供 agent 直接执行。

## 流程

### 步骤 1：获取工单

询问用户：
> Linear 工单标识符是什么？(例如 ENG-42)
> 如果你还没有创建工单，请先在 Linear 中创建一个空白工单，然后把标识符给我。

获得标识符后，使用 Linear MCP 工具获取当前工单详情（标题、描述、标签、优先级）。

### 步骤 2：理解目标

如果工单已有描述，阅读并向用户总结你的理解。询问是否有遗漏。

如果工单没有描述或描述很简单，按顺序逐个提出以下问题（每个问题等待用户回答后再问下一个）：

1. **我们要做什么？** — 用一句话描述这个功能、修复或变更。
2. **为什么？** — 这个改动解决什么问题或带来什么价值？
3. **涉及代码库的哪些部分？** — 哪些区域、包或应用会受到影响？
4. **有设计稿吗？** — Figma 链接、截图或其他视觉参考？
5. **有依赖吗？** — 这个任务是否依赖其他工单？需要 API 变更吗？
6. **什么不在范围内？** — 哪些内容明确不包含在这个工单中？

### 步骤 3：调研和上下文

根据回答，在代码库中进行针对性调研：
- 阅读即将被修改的相关现有代码
- 检查相关的文档、规范或决策记录
- 查看已完成的类似工作作为参考

总结你的发现，并与用户确认方案。

### 步骤 4：生成验收标准

根据对话，生成结构化的验收标准 JSON 块：

```json
{
  "criteria": [
    { "description": "描述必须满足的条件", "verified": false },
    { "description": "另一个需求", "verified": false }
  ]
}
```

良好验收标准的指南：
- 每个标准独立可验证 — 一个条件，不是复合语句
- 同时包含功能需求（做什么）和质量需求（测试、类型、架构）
- 始终包含：typecheck/build 通过无错误
- 始终包含：所有现有测试通过
- 如果是 UI 变更：包含设计准确性和可访问性标准
- 如果是新逻辑：包含"单元测试覆盖新逻辑"
- 优先使用"X 在移动端断点正确渲染"而不是"X 看起来不错" — 要具体

向用户展示验收标准供审核。根据反馈进行增删改。

### 步骤 5：生成工单描述

按以下格式编写结构化的工单描述：

```markdown
## 概述
[描述本工单交付内容的一段话]

## 背景
[为什么需要这个，任何相关的背景信息]

## 范围
**范围内：**
- [包含的内容列表]

**范围外：**
- [明确排除的内容列表]

## 实施说明
- [需要修改的关键文件]
- [需要遵循的相关模式]
- [任何技术注意事项或陷阱]

## 验收标准
\`\`\`json
{
  "criteria": [
    { "description": "...", "verified": false }
  ]
}
\`\`\`

## 参考资料
- [Figma、文档、相关工单、PR 的链接]
```

### 步骤 6：更新 Linear 工单

向用户展示完整的描述并请求批准。批准后：

1. 使用 Linear MCP 工具用生成的内容更新工单描述。
2. 确认更新成功。
3. 报告："工单 [identifier] 已准备好供 agent 执行。当你想要 agent 认领时，将其移动到 **Todo**。"

## 提示

- 保持标准原子化 — 每个标准只包含一个条件
- 尽可能引用具体的文件和组件
- 如果用户提到应该拆分成另一个工单的内容，记录下来但保持当前工单聚焦
- 验收标准 JSON 块是机器可读的 — Stokowski agent 被指示在移动到人工审核前验证每个标准
