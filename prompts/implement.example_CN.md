## 目标

实现解决方案，创建 PR，并确保它通过所有质量检查。

## 首次运行

1. 阅读 Linear 评论中的调查总结。
2. 阅读调查中识别的相关源文件。
3. 从 `main` 创建功能分支：
   ```
   git checkout -b {{ issue.identifier | lower }}-<简短描述>
   ```
4. 用干净的、逻辑清晰的提交实现更改。
5. 运行完整的质量套件：
   - 类型检查
   - Linting
   - 所有测试
6. 在继续之前修复任何失败。
7. 推送分支并创建 PR：
   ```
   git push -u origin HEAD
   gh pr create --title "{{ issue.identifier }}: <简洁标题>" --body "<描述>"
   ```
8. 将 PR 链接到 Linear 问题。
9. 更新工作板：做了什么、测试了什么、任何已知的限制。

## 重新处理运行

如果是重新处理运行（分支和 PR 已存在）：

1. 找到现有的 PR：
   ```
   gh pr list --head <分支名>
   ```
2. 阅读审查评论和请求的更改：
   ```
   gh pr view <编号> --comments
   ```
3. 专门解决每个反馈。
4. 再次运行完整的质量套件。
5. 推送新提交到现有分支（不要强制推送）。
6. 在 GitHub PR 上发布一条总结重新处理的评论：
   - 解决了哪些审查评论
   - 修改了什么
   - 任何决定或权衡
7. 将重新处理部分追加到 Linear 工作板。

## 完成协议

当所有工作完成且 PR 准备好审查时：

1. **发布最终完成评论** 到 Linear，包含：
   - 已实现内容的总结
   - PR 编号和链接
   - 分支名称
   - 测试结果（通过/失败计数）
   - 任何已知的限制或后续事项

2. **将 Linear 问题移动到 "Human Review" 状态** 以触发实施审查门控。

3. **发布完成评论后立即停止并退出**。不要：
   - 继续处理问题
   - 添加更多评论
   - 再次检查 PR 状态
   - 等待响应

你的会话在完成评论发布时结束。人类审查者将接管后续工作。
