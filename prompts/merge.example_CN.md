# 合并阶段

你正在合并 **{{ issue.identifier }}** 的已批准 PR：{{ issue.title }}

**URL：** {{ issue.url }}

## 目标

合并 PR 并将问题移动到最终状态。这是一个简短的、机械性的阶段 — 不需要新的代码更改。

## 流程

1. 找到这个问题的开放 PR：
   ```
   gh pr list --head <分支名>
   ```
2. 验证 PR 已批准且 CI 通过：
   ```
   gh pr view <编号> --json reviewDecision,statusCheckRollup
   ```
3. 如果 CI 失败，简单调查。如果是 flaky 测试或临时失败，重新运行检查。如果是真正的失败，在 Linear 问题上发布评论并停止。
4. 使用 squash 合并来合并 PR：
   ```
   gh pr merge <编号> --squash --delete-branch
   ```
5. 用合并确认更新 Linear 工作板。

6. **更新工作板后立即停止并退出**。不要：
   - 重新检查 PR 是否已合并
   - 发布额外的确认评论
   - 尝试移动 Linear 问题状态（编排器会自动处理）
   - 等待 CI 或人类响应

你的会话在工作板更新时结束。编排器将把问题转换到 Done。

## 重新处理运行

如果是重新处理运行（之前尝试过合并但失败了）：

1. 检查之前合并尝试失败的原因（CI 失败、合并冲突等）。
2. 如果有合并冲突：
   - 将分支 rebase 到 `main` 并解决冲突。
   - 推送更新的分支。
   - 等待 CI 通过，然后合并。
3. 如果 CI 失败：
   - 阅读失败日志。
   - 如果是 PR 更改导致的测试失败，发布详情到 Linear 并停止（这需要回到实施阶段）。
   - 如果是 flaky 或基础设施问题，重新运行并重试合并。
4. 用发生的事情更新工作板。

5. **更新工作板后立即停止并退出**。

## 不要做

- 除了冲突解决外不要做代码更改。
- 不要打开新的 PR。
- 不要跳过 CI 检查。
